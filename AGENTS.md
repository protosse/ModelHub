# Agent 工作约定（ModelHub）

面向在本仓库协作的编码 Agent / 助手。人类开发者也可参照。

## 需求文档

- **现行规格**：`docs/REQUIREMENTS.md`
- 文档写明：若代码与文档冲突，**以代码为准并回写文档**。

### 何时必须更新 `docs/REQUIREMENTS.md`

在完成实现或审查时，若识别到下列任一变化，**同一任务内**更新需求文档（不要只改代码）：

1. **产品行为**变化（用户可见流程、默认值、筛选/搜索、勾选语义、Toast、错误处理、keep-alive、草稿生命周期等）
2. **导入 / Apply / 绑定 / 测试** 等核心规则变化（覆盖语义、合并去重、写出范围、协议含义等）
3. **API / 命令 / 持久化字段** 增删改（Tauri commands、store 结构、secrets 策略等）
4. **明确非目标** 或决策清单（§1.3 / §9）需要增补或修正
5. 文档已描述的行为与当前代码不一致（以代码为准回写）

### 何时不必改需求文档

- 纯重构、性能优化、命名整理，且**用户可见语义不变**
- 仅修 bug 且文档原本已正确描述预期行为
- 测试、格式化、依赖小版本升级

### 更新方式

1. 改对应章节（功能模块、决策清单、限制、API 摘要等）
2. 视情况更新文首「最后更新」日期
3. 在 **§12 修订记录** 追加一行简明说明
4. 不写纯实现细节（组件名、memo、内部 helper），写**产品语义与对外契约**

### 冲突处理

- 代码已改、文档未改 → **补文档**
- 用户明确要求改需求但未改代码 → 先对齐需求再实现，或在文档标注「待实现」
- 不确定是否算行为变化 → **偏保守：更新文档**，或在回复中说明并征求确认

## 其它

- 技术栈与目录以 `docs/REQUIREMENTS.md` 与仓库现状为准
- 非目标范围内的能力（代理、余额、Keychain 等）不要擅自做成「默认产品能力」；若要做，先改需求文档再实现
